## 1. Design Brief

### Emotional Targets

Focused command, accountable momentum, restrained nostalgia.

The interface should make a long autonomous development run feel controlled and
observable. Retro styling is a mood layer, not a toy layer: every visible
element must help the operator understand state, risk, or next action.

### Aesthetic Identity

- **Reference domain**: Mission-control terminal benches, service monitors,
  and late mainframe operator consoles.
- **Era / movement**: Late-1970s and early-1980s CRT computing interpreted
  through Swiss information hierarchy.
- **Material metaphor**: Phosphor text behind smoked glass inside a black metal
  instrument rack.

The `EXAMPLE/cool-retro-term` reference supports profile-driven CRT moods,
terminal framing, scanline/noise categories, settings groups, and render-stage
thinking. It must remain reference-only: do not copy QML, shader code, shader
constants, profile data, colors, images, icons, fonts, resource manifests, or
build scripts.

### Signature Moment

The screenshot-worthy moment is the iteration frame: a compact boot/status
header locks the project, provider, model, theme, iteration, elapsed time,
current operation, and dry-run state into one readable strip; beneath it,
manager decision, prompt preview, live Codex execution, response summary, and
DB log confirmation advance as a single operational cycle.

### Micro-Narrative

Arrival -> Orientation -> Engagement -> Action -> Resolution

```
Launch/config -> Boot panel -> Iteration frame -> Manager decision
             -> Live execution -> Result/log -> Next command or stop
```

The user should feel that the system is booting an autonomous instrument,
establishing context, acting visibly, recording its work, and handing off to the
next command without drama.

## 2. User Flows

### Flow 1: Start Or Resume Autonomous Run

**Trigger**: Operator runs `src/apex_infinite/cli.py` with `--path` or starts
interactive selection.
**Goal**: Start the Apex Spec loop with clear context and a visible first
iteration.

```
CLI launch -> Config load -> Project path resolution -> Boot panel
          -> History summary -> Manager decision -> Prompt preview
          -> Codex execution -> DB log -> Next iteration or stop
```

**Happy path**: The operator sees provider, model, project path, max
iterations, start command, CEO instruction presence, selected theme, and fallback
mode before the first decision runs.
**Error states**: Missing config, unknown provider, invalid UI theme, invalid
project path, missing Codex binary, provider retry exhaustion, and max-iteration
stop must render as unmistakable labeled states in styled and plain modes.

### Flow 2: Dry-Run Confidence Check

**Trigger**: Operator runs with `--dry-run`.
**Goal**: Verify provider/model, project path, command routing, UI mode, and the
exact Codex command without launching Codex.

```
CLI launch -> UI/config resolver -> Manager/start decision
          -> Prompt preview -> Dry-run command panel -> Raw history log
```

**Happy path**: Dry-run output clearly says no Codex subprocess was launched and
shows the command that would run.
**Error states**: Invalid config, invalid theme, and missing project path fail
before the dry-run command is displayed.

### Flow 3: Monitor Long Codex Execution

**Trigger**: Manager chooses a known Apex Spec command or custom instruction.
**Goal**: Keep the operator aware while `codex exec` runs and preserve captured
output semantics.

```
Decision -> Prompt build -> Codex status start -> Elapsed-time status
        -> Success | stderr-only | non-zero exit | timeout | missing binary
        -> Response panel -> DB log
```

**Happy path**: A live status line or equivalent display shows command state,
elapsed time, timeout threshold, and current operation without leaking styled
output into SQLite history.
**Error states**: Timeout, non-zero exit, missing binary, generic exception,
and interrupted run must have severity labels, plain text fallbacks, and DB rows
containing raw operational facts only.

### Flow 4: Inspect History Ledger

**Trigger**: Operator runs `--history`, optionally with `--path`.
**Goal**: Read recent workflow state without horizontal scrolling.

```
History flag -> Optional path filter -> Compact ledger
             -> Verbose expansion when --verbose is present
```

**Happy path**: Default history uses ID, timestamp, project, manager output,
status/help marker, concise reason, and truncated response summary. Verbose
history expands reason and response detail.
**Error states**: Empty history and legacy rows with sparse values render as
clear text, not broken tables.

### Flow 5: Emergency Help Or Operator Interrupt

**Trigger**: Manager emits `help`, or the operator presses `Ctrl+C`.
**Goal**: Pause visibly without corrupting loop state.

```
Help/interrupt -> Alert block -> Desktop notification when available
               -> Operator input or quit -> Continue/log or stop
```

**Happy path**: The prompt makes clear whether the system is waiting for input,
what reason triggered the pause, and whether the response will be used as CEO
instructions.
**Error states**: A second `Ctrl+C` force quit remains possible and visibly
different from a normal pause.

### Flow 6: Emit Events For Future Visual Surfaces

**Trigger**: Operator passes `--event-stream PATH`, or a future wrapper starts
the CLI as a subprocess.
**Goal**: Give wrappers raw lifecycle facts without scraping human output.

```
CLI launch -> Event emitter open -> Lifecycle JSONL writes
          -> Human Rich/plain output remains separate
          -> Wrapper consumes events -> Wrapper log/status panels
```

**Happy path**: Events include iteration lifecycle, manager decision, prompt
dispatch, subprocess lifecycle, output summary, DB log, help, completion,
timeout, and errors.
**Error states**: `--event-stream -` cannot mix JSONL with Rich or plain human
output; the CLI must require an explicit human-rendering disable mechanism.

## 3. Screen Inventory

For this CLI, "screen" means a distinct terminal surface, command output mode,
or future wrapper view.

| Screen | Route/Path | Purpose | Key Components |
|--------|------------|---------|----------------|
| Interactive Project Selection | `apex-infinite` | Choose project, start command, and CEO instruction when flags are absent | Project list, text prompt, validation error |
| Boot Status Panel | Normal run start | Establish operational context before iteration 1 | Project, provider/model, theme, fallback mode, max iterations, dry-run, start command |
| Iteration Frame | Main loop | Hold each autonomous cycle together | Iteration number, status strip, history count, current operation |
| Manager Decision Panel | Main loop after LLM decision | Separate manager choice from Codex output | Output command, reason, known/custom routing label |
| Prompt Preview | Before Codex execution | Show what will be sent to Codex without overwhelming output | Truncated prompt, known-command badge, custom instruction badge |
| Codex Execution Status | During subprocess | Show active process and elapsed time | Spinner/status text or static fallback, timeout threshold, binary name |
| Agent Response Panel | After subprocess | Summarize captured agent output | Truncated response, verbose expansion, error state label |
| DB Log Notice | End of iteration | Confirm durable history write | Project key, command, timestamp/status |
| Help Pause | `help` decision | Pause for operator input | Alert label, reason, input prompt, notification |
| Interrupt Pause | `Ctrl+C` once | Collect new CEO instruction after current step | Alert label, input prompt, quit option |
| Completion State | `alldonebaby` | End the workflow visibly | Completion message, reason, iteration count, history marker |
| Max Iteration Stop | Safety limit reached | Stop runaway loops safely | Warning, max count, next operator action hint |
| History Ledger | `--history` | Review previous interactions | Compact ledger, path filter, verbose detail |
| Config Error Surface | Startup/config load | Fail fast for invalid user configuration | Error message, source field, valid values |
| Event Stream | `--event-stream PATH` | Machine-readable lifecycle output | JSONL event names, raw payload facts, no Rich markup |
| Linux Wrapper Main Window | Optional future mode | Render event-driven visual console | Read-only log viewport, status panels, theme selector, effect controls |
| Linux Wrapper Settings | Optional future mode | Configure visual-only wrapper preferences | Theme, effect level, font/scaling, reduced effects, plain fallback |

## 4. Navigation Structure

```
src/apex_infinite/cli.py
|-- Normal run
|   |-- Interactive prompts when --path is absent
|   |-- Boot status panel
|   |-- Iteration loop
|   |   |-- History summary
|   |   |-- Manager decision
|   |   |-- Prompt preview
|   |   |-- Codex execution
|   |   |-- Response panel
|   |   \-- DB log
|   |-- Help or interrupt pause
|   \-- Completion or max-iteration stop
|-- Dry run
|   \-- Same loop with no Codex subprocess launch
|-- History
|   \-- Compact ledger, optional verbose expansion
|-- Event stream
|   \-- JSONL side channel for wrappers/tests
\-- Optional Linux wrapper
    |-- Main visual console
    \-- Wrapper settings
```

**Navigation pattern**: CLI flags determine entry mode. Inside normal runs, the
only interactive navigation is the initial project selection and emergency
help/interrupt input.
**Deep linking**: Project history is scoped by normalized project path. Event
streams are addressed by file path. Future wrapper views must consume event
names instead of scraping Rich terminal frames.

## 5. Interaction Patterns

### Forms

- Validation: Fail fast at Click/config boundaries before loop execution.
- Error display: One visible severity label, the invalid field/value, and the
  accepted values or environmental reason.
- Success feedback: Boot panel confirms resolved provider/model/theme/project
  before work begins.

### Modals/Dialogs

- Base CLI: no modal dialogs. Use terminal prompts only for project selection,
  `help`, and interrupt responses.
- Future wrapper: dialogs are acceptable only for settings import/export,
  destructive reset, or file selection. Normal workflow state must stay in the
  main console.

### Loading States

- LLM calls: show summarizer/manager operation and retry count.
- Codex subprocess: show elapsed time and process state while preserving
  stdout/stderr capture behavior.
- Plain mode: use static, line-oriented progress messages instead of live-only
  renderers or carriage-return animation.

### Notifications

- Terminal bell and `notify-send` remain support channels for help,
  completion, and interrupt states.
- Notifications must never be the only place an important state appears.
- CI, non-TTY, `NO_COLOR`, `TERM=dumb`, and redirected output use plain text
  with no ANSI escapes or box glyphs.

### Product Surface Boundaries

- Primary surfaces: boot panel, iteration frame, manager decision, prompt
  preview, execution status, response panel, history ledger, pause states, and
  completion states.
- Developer/admin/debug surfaces: pytest output, renderer fixture snapshots,
  raw event JSONL files, config validation details, and wrapper spike logs.
- Excluded from primary UI: renderer internals, route ownership notes,
  frame/glyph diagnostics, visual test scaffolding labels, raw JSON event dumps,
  shader reference details, and copied reference profile metadata.

## 6. Motion and Animation Strategy

### Philosophy

Motion should communicate liveness and time, never distract from commands,
errors, or handoffs.

### Entrance Choreography

- Page load: Terminal output should appear in stable sections: boot panel,
  iteration header, decision, prompt, execution, result, log.
- Scroll reveals: None in the base CLI. Future wrapper may use subtle section
  arrival when new events are received.

### Interaction Feedback

- Hover states: Not applicable to base CLI. Future wrapper may use restrained
  hover highlights on settings and copied/log actions.
- Click/tap responses: Not applicable to base CLI. Future wrapper controls
  should use immediate visual state and text labels.
- Focus rings: Terminal prompts use clear text labels. Future wrapper focus
  states must be visible in high-contrast themes and plain fallback.

### Scroll-Driven Moments

- Base CLI: no scroll-driven animation.
- Future wrapper: viewport scroll should keep active status pinned while the log
  remains readable and selectable.

### Animation Constraints

- Target locked 60fps for live terminal updates and wrapper effects.
- In the base CLI, live displays must degrade to line-oriented messages under
  `--plain`, `--compact`, `NO_COLOR`, `TERM=dumb`, non-TTY output, redirected
  output, or `Console().is_terminal == false`.
- Maximum 3 simultaneous animated elements per viewport region in the wrapper.
- Prefer low effect intensity by default. Reduced-motion mode keeps status
  changes visible through text, color, or simple opacity changes.
- Do not animate over error, timeout, help, interrupt, or completion text.

## 7. Layout Philosophy

### Composition Approach

Dense and information-rich, with enough rhythm to separate phases of work. The
base CLI should feel like one instrument panel, not a set of unrelated print
statements.

### Visual Hierarchy

- Scale contrast: Use labels, borders, and color roles more than large text.
  Important states get concise uppercase labels only when they need immediate
  attention.
- Negative space: Default mode may use one blank line between major sections.
  Compact mode removes decorative spacing while preserving labels.
- Section rhythm: Every iteration uses the same order so operators can scan by
  muscle memory.

### Section Transitions

Use hard terminal section boundaries: boot, iteration, decision, execution,
result, log, next step. Rich themes may use faint repeated separators for a
low-fidelity scanline feel; plain and compact modes use simple ASCII labels.

## 8. Responsive Strategy

| Breakpoint | Target | Layout Approach |
|------------|--------|-----------------|
| `< 80 columns` | Narrow terminal/logs | Single-column labels, no wide tables, no hidden errors, truncate only with explicit counts |
| `80 columns` | Baseline terminal | Compact status strip, wrapped reason text, readable history ledger |
| `100 columns` | Comfortable terminal | Full iteration frame, concise panels, response summaries |
| `120+ columns` | Wide terminal | Wider response/history columns without adding decorative noise |
| `non-TTY/log` | CI, redirected output, dumb terminal | Plain line output, no color, no live rendering, no box glyphs |

**Approach**: Terminal-width adaptive, with plain-output fallbacks ahead of
visual polish.
**Touch targets**: Future wrapper controls must be at least 44x44px. Base CLI
interactive prompts must remain keyboard-first.

## 9. Accessibility

**Target**: WCAG 2.1 AA intent for contrast and non-color semantics, adapted to
terminal constraints.

- Keyboard navigation: All base CLI interactions must be possible from the
  keyboard. Future wrapper controls require normal tab order and visible focus.
- Screen reader: Plain mode should produce sequential, meaningful text without
  relying on Rich layout. Live statuses must also emit durable text transitions.
- Color contrast: Styled themes must keep normal text, muted text, warnings,
  errors, success, and borders readable on common dark terminals.
- Focus management: Help and interrupt prompts must clearly own input focus.
- Reduced motion: `--plain`, `--compact`, `ui.effect_level: off`, `NO_COLOR`,
  `TERM=dumb`, non-TTY output, and redirected output must remove live-only or
  decorative effects while preserving state changes.
- Non-ASCII: `--ascii` and plain mode must use code points 0-127 only.
- Color independence: Every status needs a text label in addition to color.

## 10. Design System

### Color Architecture

- **Dominant surface (60%)**: Terminal background or wrapper viewport. Default
  interactive theme should feel dark, quiet, and high contrast.
- **Secondary surfaces (25%)**: Panels, iteration frames, response blocks, and
  history rows. Use low-contrast borders or muted color, not heavy decoration.
- **Accent (10%)**: Active command, current operation, elapsed time, selected
  theme, and focused prompt.
- **Signal colors (5%)**: Success, warning, error, timeout, help, interrupt,
  dry-run, and completion states.

Palette character: Synthetic, dark, restrained, and high-contrast. Use
independent token names and values. Do not copy `cool-retro-term` profile
colors.

Built-in theme intents:

| Theme | Intent | Primary Use |
|-------|--------|-------------|
| `auto` | Resolve to an appropriate default for the terminal | First-run default |
| `crt-green` | Focused green-phosphor operator console | Capable interactive TTY default |
| `crt-amber` | Warm amber maintenance terminal | Operator preference |
| `ibm-dos` | Crisp DOS-like contrast with minimal effects | High-legibility retro mode |
| `plain` | Deterministic unstyled text | Logs, CI, non-TTY, dumb terminals |

### Typography

- **Display font**: The user's terminal font in bold or high-emphasis style.
  Do not ship copied or bundled reference fonts in the base CLI.
- **Body font**: The user's terminal monospace font.
- **Monospace**: Primary typography for every base CLI surface.
- **Scale ratio**: Terminal emphasis scale: normal, dim, bold, panel title,
  alert title. Do not depend on viewport-width font scaling.
- **Minimum body size**: Controlled by terminal; future wrapper should default
  to a readable system monospace size and expose scaling controls.

### Spacing Scale

Terminal spacing uses rows and columns:

```
0 rows: compact inline labels
1 row: default section separation
2 rows: major state transition
4 columns: nested detail indent
8 columns: table padding or status grouping when width allows
```

Future wrapper spacing should use an 8px base scale: 4, 8, 12, 16, 24, 32,
48, 64.

### Elevation and Depth

Base CLI depth comes from ordering, borders, labels, and muted text. Do not
simulate physical depth with heavy boxes inside boxes. Future wrapper may use
subtle glow, dark glass, and frame treatment, but not at the expense of text
legibility.

### Texture and Atmosphere

Rich milestone:
- Low-fidelity scanline-like separators may use simple repeated characters.
- Disable decorative separators in `plain` and compact modes.
- Never write box glyphs, ANSI escapes, or Rich markup to SQLite history.

Future wrapper:
- Effects may include independently implemented glow, scanlines, flicker,
  curvature, phosphor trail, and frame treatment.
- Do not use copied noise images, CRT images, shaders, constants, fonts, or
  resource manifests from `EXAMPLE/cool-retro-term`.

## 11. Component Patterns

| Component | Used In | Behavior |
|-----------|---------|----------|
| UI Config Resolver | Startup | Resolves CLI flags, environment constraints, config, and defaults deterministically |
| Theme Token Set | Renderer, wrapper | Supplies color, severity, border, glyph, and fallback tokens without copying reference data |
| Glyph Set | Renderer | Switches between styled, ASCII, compact, and plain symbols |
| Boot Panel | Normal and dry-run startup | Shows provider, model, project, theme, max iterations, dry-run, start command |
| Status Strip | Iteration frame | Shows iteration, operation, elapsed time, provider/model, project, dry-run |
| Iteration Frame | Main loop | Groups one autonomous cycle with stable section order |
| Decision Panel | Manager output | Separates command/custom instruction and reason from agent output |
| Prompt Preview | Before Codex | Shows generated Codex prompt with truncation and command classification |
| Execution Status | Codex subprocess | Shows process state, elapsed time, timeout, and binary name |
| Response Panel | After Codex | Shows captured output summary, verbose expansion, and error labels |
| History Ledger | `--history` | Shows compact rows, path filter, verbose expansion, empty state |
| Alert Block | Help, interrupt, timeout, error | Uses text labels plus color/severity; impossible to miss in plain mode |
| DB Log Notice | End of iteration | Confirms history write without exposing implementation diagnostics |
| Event Emitter | Event stream | Writes raw JSONL facts and exposes importable API for tests/wrappers |
| Wrapper Log Viewport | Future Linux wrapper | Read-only event-driven operational log, not a terminal emulator clone |
| Wrapper Settings Panel | Future Linux wrapper | Theme, effect intensity, font/scaling, reduced effects, plain fallback |

## 12. Anti-Patterns to Avoid

- Do not copy `cool-retro-term` source, shader code, shader constants, profile
  JSON, literal colors, images, icons, fonts, resource manifests, or build
  scripts.
- Do not make the base CLI require Qt, PySide6, PyQt, qmltermwidget,
  QTermWidget, a graphical display, or an external terminal emulator.
- Do not let visual styling change manager prompts, command normalization,
  path normalization, timeout behavior, DB schema compatibility, or
  autonomous `Next command:` handoffs.
- Do not hide errors, timeouts, help pauses, interrupts, non-zero exits, or
  completion states in compact or styled output.
- Do not use wide history tables that require horizontal scrolling at
  80 columns.
- Do not store styled output, ANSI escapes, Rich markup, frame glyphs, or
  visual tokens in SQLite rows or JSONL events.
- Do not turn primary operator surfaces into debug dashboards. Renderer tests,
  raw event payloads, and frame diagnostics belong in developer surfaces.

## 13. Assumptions And Conflict Resolutions

### Working Assumptions

- Apex Infinite CLI is terminal-first and operator-facing, not a web or
  marketing surface: `PRD.md`, `README.md`,
  and `src/apex_infinite/cli.py` all describe a Python CLI around manager LLM decisions,
  Codex subprocess execution, SQLite history, and terminal output. This is safe
  because the UX PRD defines terminal surfaces and optional wrapper surfaces
  without introducing unrelated app screens.
- The first visual milestone should use Rich only: the current dependency set
  already includes Rich, the upgrade plan prioritizes a Rich console before any
  wrapper, and base CLI portability is a PRD requirement. This is safe because
  wrapper-only effects are deferred to the event-stream milestone.
- `auto` should resolve to `crt-green` only for capable interactive terminals
  and to `plain` for constrained output: this is stated in `PRD.md`,
  `CONVENTIONS.md`, and the upgrade plan. This is safe because invalid user
  theme configuration still fails fast instead of silently falling back.
- The example repository informs UX vocabulary but not implementation data:
  its settings files show profile management, font/scaling controls, effect
  toggles, performance controls, menus, tabs, frame treatment, and render-stage
  concepts; its license and the project plan forbid copying. This is safe
  because the PRD records clean-room boundaries as product requirements.
- Operational telemetry is part of the product only when it helps the operator:
  status strip, elapsed time, timeout, provider/model, manager decision, prompt
  preview, DB log confirmation, and history ledger are primary surfaces because
  the product is an autonomous workflow runner. Renderer internals and raw event
  payload dumps remain developer/debug surfaces.

### Conflict Resolutions

- The existing phase tracker file names Phase 00 "Foundation", while
  `state.json` and `PRD.md` name Phase 00 "CLI Operator Console MVP". The UX
  PRD follows `state.json` and `PRD.md` because they are the current functional
  planning sources for createuxprd output.
- The functional PRD lists 5 sessions for Phase 00 and 2 for Phase 01, while
  the upgrade plan includes a more detailed 8-session split with release
  verification. The UX PRD resolves this by defining product surfaces and UX
  requirements across the full upgrade while leaving exact session count to
  `phasebuild`.
- `cool-retro-term` is a full graphical terminal emulator with tabs, menus,
  settings dialogs, shader passes, copied assets, and external submodules,
  while Apex Infinite CLI is a Python workflow manager. The chosen
  interpretation is clean-room visual translation: Rich gets status hierarchy,
  framing, profile-like themes, and low-fidelity separators; the optional Linux
  wrapper later gets event-driven visual effects without depending on
  `cool-retro-term` or qmltermwidget.
